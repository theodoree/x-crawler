**新工具/技术**  
- bun v1.3.11 — 建议用 bun upgrade（而不是 bun update）来升级运行环境，避免环境差异导致的问题。  
- pi-mono — 轻量级 AI agent 库，作者建议若只是做 agent，这类库比研究完整 Claude/Codex 源码更合适。  
- Claude Code（通过 source-map 还原的源码片段）— 泄露出的源码能暴露协议与防护逻辑，但缺少私有包和完整脚手架。  
- plain-crypto-js（恶意 npm 包，出现在 axios 投毒事件中）— 典型的供应链载荷/loader，用来动态解码并执行恶意命令。  

**核心观点/方法论**  
- 尽早 fork/下载泄露仓库以便分析 — 源码可能很快被下架或变动，但需注意法律与伦理边界。  
- 混淆还原后的 JS 有限价值 — 大规模（示例：50 万行）混淆代码即便能还原，仍缺注释、模块关系与工程结构，完整恢复并运行成本极高。  
- 反蒸馏三层策略（文章总结）：第一层：在输出里掺假，污染被抓取用于训练的样本，降低对手蒸馏质量。  
- 第二层：隐藏中间推理，用摘要+签名替换推理链，客户端持签名换回原文，外部观察者看不到完整推理。  
- 第三层：协议隔离，使用带版本标记的专用 JSON 协议将特殊请求与普通 API 流量隔离，既防伪装也能节省输出 token。  
- 产品与源码取舍：Codex 虽然开源但 CLI 难用；研究源码价值需权衡成本，常见场景用轻量库更高效。  

**实践经验/案例**  
- 实操案例：dotey 成功把部分还原的 Claude Code 跑起来，遇到小 bug 后把错误信息发给 Codex 就解决，说明社区协作能快速推动可复现性。  
- 安全案例：axios 被投毒（恶意版本如 axios@1.14.1），提醒要立即 pin 版本并审计 lockfile，应对 npm 供应链攻击。  
- 环境建议：运行泄露/复现工程前先统一 runtime（如升级 bun 到 v1.3.11），能避免不必要的环境错误。  
- 其他观察：社区还复现了泄露代码里的互动/愚人节功能（buddy 伴侣精灵），展示源码可揭示未发布功能与交互设计。
