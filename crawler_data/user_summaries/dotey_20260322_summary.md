**新工具/技术**（如有）
- baoyu-youtube-transcript：一个能直接抓取 YouTube 字幕并输出带章节、说话人标注和封面图的 skill，支持 SRT 和带时间戳的 Markdown。
- 实现要点：直接调用 YouTube InnerTube（无需 Google API Key/OAuth），支持完整链接/短链/嵌入/Shorts/视频 ID 与多语言。
- 输出与缓存：默认 Markdown（可导出 SRT），首次抓取缓存 meta.json、transcript-raw.json、transcript-sentences.json、cover.jpg，后续秒级重排无需再请求网络（可用 --refresh 强制刷新）。
- 说话人识别：启动 AI 子代理（用 Claude Sonnet），按元数据→对话线索→通用标签三层优先级识别并能回溯更新，输出分段对话每段带时间范围。
- 可与 baoyu-translate 和多种 agent（如 Claude Code、小龙虾）配合使用；安装命令提供 npx 一键添加，项目增长数据：10K+ stars、约 193.7K installs。

**核心观点/方法论**（如有）
- 无需官方 API 的实用主义：直接利用公开但未文档化的内部接口，降低接入门槛与成本，便于快速产出工程化工具。
- 面向“可读文本”而非严格字幕的设计选择：通过智能断句（按标点合并、跨片段合并、按字符比分配时长）提高阅读连贯性，但会牺牲部分时间轴精度。
- 多层优先级与回溯机制解决不完备元信息：将元数据推断与语义线索结合，保证说话人标签的一致性和可纠错性。
- 缓存优先的用户体验优化：第一次抓取即全量缓存，后续切格式或调整参数秒级出结果，减少重复网络开销与延迟。
- 管理+拆解是驾驭 Agent 的核心能力：技术实现之外，定义问题、拆分任务与知人善用比单纯编码更重要。

**实践经验/案例**（如有）
- 时间轴/字幕精度权衡：导出的原始 SRT 时间戳很准，但合并后按字符估算时长会导致段落时间不精确；若需严格对齐建议用 whisper 等需加载模型的方案。
- 使用要点：用 --speakers 启动说话人识别流程；首次抓取后利用缓存快速切换输出格式；遇变动可加 --refresh 强制重新抓取。
- 工具链组合价值：和 baoyu-translate 联用，可一键抓取并高质量翻译播客/视频稿，适合产出可读笔记、翻译稿与节目文档。
- 开发与交付建议（来源于转译的 Codex 指令）：在交付前严格自测、定义完成标准并用简单直白的语言汇报结果，能显著减少迭代成本。
