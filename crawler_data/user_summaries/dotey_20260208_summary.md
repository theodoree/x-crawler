**新工具/技术**
- “Build Any App: The Technical Co-Founder” 提示词模板：把产品从需求到交付的流程塞进一个长提示词，适用于从零构建完整产品的场景，但会占用上下文窗口。  
- code agent / LLM 自动生成单元测试与代码补全：把 AI 用作补写单元测试和修复遗留代码的工具，能快速覆盖和验证行为。

**核心观点/方法论**
- TypeScript 能显著降低返工率（作者估计 JS 返工率 ~50%，TS ~30%），因此即便增加行数/tokens，也能节省总体时间和成本。  
- 轻量提示词优先：一句话目标 + 必要背景 + 约束 + 期望输出格式，日常对话不用把大型模板当默认上下文，避免稀释有用信息。  
- 模板按需使用：结构化长提示词适合完整产品引导，不适合日常小任务或频繁对话（会污染上下文）。  
- 对 AI 代码的审查策略：可以减少逐行审查，更多给方向性建议并让 AI 自我验证，因为 AI 经常能把各方面考虑周全。

**实践经验/案例**
- AI 补单元测试很高效：对于“屎山”遗留代码，AI 生成单元测试并进行多次验证，通常比招新手更快上手并覆盖问题路径。  
- 大型单体项目对 code agent 限制大：业务耦合复杂、影响面广时，AI 容易出错，需要把系统拆小（高内聚低耦合 / 微服务化）以降低风险。  
- 模型已经学会很多好实践：不必把常见做法反复写进提示词，直接用清晰需求，模型会做澄清、分步骤和关键节点确认。  
- 成本权衡提示：TypeScript 会增加代码量（小型 +10–30%，中大型 +20–60% 行数/token），要在 token 成本与减少返工带来的时间节约之间权衡。  
- 实操建议：平时用轻量描述，遇到需要逐步引导的完整项目再用结构化模板；在让 AI 改代码时要求它同时给出验证用例或测试以保证安全。
